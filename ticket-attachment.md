## 1. Симптомы

С разных устройств в РФ-сети (МТС, Билайн, домашний Wi-Fi разных провайдеров) периодически наблюдаются:

- TLS handshake обрывается по таймауту для всех HTTPS-сайтов на VPS:
  - `zavod.smm-ministr.ru`
  - `survey.smm-ministr.ru`
  - `brand.smm-ministr.ru`
  - `krutly.com`
- При SSH-подключениях: `kex_exchange_identification: read: Connection reset by peer`
- Восстановление само через несколько минут (около 5)
- Помогает переключение на VPN с выходом в другой стране → подозрение на сетевую фильтрацию или нестабильность пиринга на РФ-стороне

## 2. Методика наблюдения — три независимые точки

Чтобы зафиксировать события с конкретными временными метками, развёрнут мониторинг с трёх независимых точек:

| Точка наблюдения | Откуда работает | Частота опроса | Где лежат логи |
|---|---|---|---|
| **Изнутри VPS** | health-check скрипт по cron на самом VPS | каждые 2 мин | `/var/log/pleyada-health.log` |
| **Снаружи, не-РФ** | GitHub Actions (Microsoft Azure DC, Virginia/Ireland) | каждые 5 мин | https://github.com/lukonin33/vps-monitor/blob/main/logs/probes.csv |
| **Снаружи, РФ** | Yandex Cloud Function (ru-central1, Новосибирск) | каждые 5 мин | https://github.com/lukonin33/vps-monitor/blob/main/logs/probes-ru.csv |

Все логи в формате CSV доступны публично без авторизации (репозиторий открытый, GitHub Pages).

Все временные метки далее даны в UTC. Для удобства соответствие часовых поясов:
- **UTC** 18:06:14 = **МСК** (Москва, UTC+3) 21:06:14 = **НСК** (Новосибирск, UTC+7) **2026-05-11 01:06:14** (переход на следующие сутки)

| Точка наблюдения | Время (UTC) | Состояние |
|---|---|---|
| РФ (Yandex Cloud) | 18:01:13 | Все 4 сайта OK + TCP 22 OK + TCP 443 OK |
| **РФ (та же точка)** | **18:06:14** | **Таймаут для всех 4 HTTPS-сайтов + TCP 22 = FAIL + TCP 443 = FAIL** |

**Что это означает:** в момент, когда точка из РФ-сети не могла установить TCP-соединение к 5.129.207.254 на портах 22 и 443, точка из не-РФ-сети через 3 минуты в том же временном окне видела VPS полностью доступным. Значит **VPS физически не падал** (работал и отвечал на запросы из не-РФ сегмента), но **сетевой путь из РФ к IP 5.129.207.254 был временно недоступен** около 5 минут.

## 6. Гипотезы

1. **🔴 Основная — нестабильность пиринга / граничного оборудования.** Hop 2 (212.164.50.113) показывает всплески до 4 секунд регулярно. Это похоже на проблему пиринга Timeweb с вышестоящим провайдером или нестабильность граничного маршрутизатора. **Это ваша сторона.**
2. **Сетевая фильтрация на стороне РФ-провайдера / ТСПУ** — менее вероятно: MTR показывает задержку, а не потерю пакетов (loss% = 0).
3. **Внутреннее сетевое событие Timeweb** — что-то на стороне инфраструктуры новосибирского ДЦ.
